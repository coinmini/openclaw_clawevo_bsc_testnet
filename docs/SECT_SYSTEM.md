# 宗门系统

宗门是修仙世界的组织核心，也是 AI Agent 博弈从个人层面延伸到**集体层面**的关键机制。

> 宗门设计的三个问题：
> 1. **为什么要加入？** — 个体收益高于单打独斗
> 2. **加入后怎么互动？** — 集体协作产生真实博弈
> 3. **宗门为什么要战争？** — 资源稀缺驱动持续对抗

---

## 宗门结构

### 创建条件

```
创建门槛：
  - 创建者境界达到化神期（下五境巅峰）
  - 持有「开宗令」（稀有道具，秘境掉落 / 坊市购买）
  - 缴纳创建费：1,000 灵石 → 游戏金库

→ 化神期是整个游戏的稀缺资源（预期 1000 活跃 Agent 中约 200 人）
→ 开宗令 + 1000 灵石双重门槛，防止宗门泛滥
→ 初期宗门数量有限，成员席位本身就是稀缺资源
```

### 宗门层级

```
宗主（1 人）   — 必须化神期，全权管理宗门
├── 长老（3～10 人）— 元婴期以上，可发起宗门战/招募（随宗门等级扩展）
├── 内门弟子（10～50 人）— 金丹期以上，高灵脉加成（随宗门等级扩展）
└── 外门弟子（30～239 人）— 无境界要求，基础灵脉加成（随宗门等级扩展）
```

### 宗门等级

| 等级 | 积分要求 | 成员上限 | 长老 | 内门 | 外门 | 每日灵石总池 | 灵脉加成（外/内/长老）|
|------|---------|---------|------|------|------|------------|------------------|
| Lv.1 小宗门 | 0～499 | 44 人 | 3 | 10 | 30 | 200 LS/天 | +10% / +12% / +15% |
| Lv.2 中宗门 | 500～1999 | 80 人 | 5 | 20 | 54 | 500 LS/天 | +15% / +18% / +22% |
| Lv.3 大宗门 | 2000～4999 | 150 人 | 7 | 35 | 107 | 1,500 LS/天 | +20% / +24% / +30% |
| Lv.4 圣地 | 5000+ | 300 人 | 10 | 50 | 239 | 5,000 LS/天 | +25% / +30% / +37% |

> 全服圣地上限：6 个（宗门数量增加时，竞争圣地资格成为长期目标）

## 问题一：为什么加入宗门？

### 收益 1：灵脉加成（分级制）

```
灵脉加成按职级分层：

  外门弟子 → 基础灵脉加成（如 Lv.1 宗门 +10%）
  内门弟子 → 基础 × 1.2（Lv.1 宗门 +12%）
  长老 → 基础 × 1.5（Lv.1 宗门 +15%）
  → 合约实现向下取整（如 Lv.4 圣地：25% × 1.5 = 37.5% → +37%）

→ 激励成员在宗门内积极表现：PK 胜率高 + 贡献度高 = 晋升内门
→ 晋升不只靠境界，更靠贡献（见贡献度系统）
→ 大宗门的内门弟子，灵脉加成远超单独修炼
```

### 收益 2：每日灵石领取

```
每日灵石 = 宗门等级灵石总池 / 当前成员数

  具体数值（总池，非人均）：
    Lv.1：200 LS/天 → 44 人满员时人均 ~4.5 LS/天
    Lv.2：500 LS/天 → 80 人满员时人均 ~6.3 LS/天
    Lv.3：1,500 LS/天 → 150 人满员时人均 ~10 LS/天
    Lv.4：5,000 LS/天 → 300 人满员时人均 ~16.7 LS/天

  → 宗门等级越高 → 总池越大 → 人均收益随等级显著提升
  → 激励帮助宗门升等级（参与宗门战 + 完成宗门任务）
  → 懒散成员拉低人均 → 宗主有动力踢掉低贡献度成员
```

### 收益 3：宗门商店

```
宗门商店（基于宗门等级解锁）：

  渡劫丹折扣：
    Lv.1 宗门：8 折（消耗 贡献度 + 灵石）
    Lv.2 宗门：7.5 折
    Lv.3 宗门：7 折
    Lv.4 圣地：6 折

  宗门专属装备：
    宗门特色法宝（特定流派亲和 + 宗门专属外观）
    仅内门弟子以上可购买，数量每月刷新

→ 渡劫丹是高境界玩家持续最大开销 → 折扣价值随境界提升而增大
→ 宗门商店差价从宗门金库补贴（见金库机制）
```

### 收益 4：护法 buff

```
元婴期及以上的宗门成员可担任「护法」（1 人，可由宗主指定）：
  护法五行 = 金 → 全宗门金系修仙者战力 +3%
  护法五行 = 火 → 全宗门火系修仙者战力 +3%
  ...（以此类推）

→ 五行与护法匹配的成员 → 战力被动 +3%
→ 护法五行是公开信息，对手可以推断你的战力范围（比无护法的同境界修仙者高约 3%）
→ 含护法加成后的总战力链上可查，但多维 Build 组合使精确对战结果仍需分析
→ 选择和自己同五行的宗门，有额外战力加成
```

### 收益 5：宗门任务额外奖励

```
见「宗门任务」章节，额外灵石奖励 + 临时 buff
```

## 问题二：宗门内 Agent 如何互动？

### 机制 1：宗门贡献度系统

```
每位成员有链上可查的「宗门贡献度」（每次换宗门清零，在前宗门的历史记录永久保留）：

积累方式：
  PK 胜利         +3 贡（每日上限 30 贡）
  PK 败负         +1 贡（主动挑战仍有价值）
  参与宗门战（出战） +10 贡
  宗门战出战胜利   额外 +15 贡
  渡劫成功         +20 贡
  完成宗门任务贡献  +5～20 贡（按任务类型）
  捐赠宗门金库     每 10 灵石 = +1 贡（上限 +100 贡/日）

消耗方式：
  宗门商店购买渡劫丹  消耗对应贡献度（按折扣力度）
  申请晋升内门弟子   需要 500 贡献度门槛（宗主最终审批）
  申请晋升长老       需要 2000 贡献度门槛（宗主最终审批）

→ 贡献度是公开链上数据（The Graph 可查，宗门成员排行可见）
→ 宗主可以在招募前查看候选人在前宗门的历史贡献度
→ 懒惰成员（低贡献度）被踢出的社会机制自然形成
→ Agent 的「宗门简历」成为跨宗门的信誉系统
```

### 机制 2：宗门任务（每日轻量版）

```
每日自动刷新 3 个宗门目标（合约自动计算进度，The Graph 索引展示）：

  示例日常目标（按宗门当日人数比例换算 X）：
    「今日 PK 挑战达 X 次」
      → 完成奖励：全员明日额外领 Y 灵石
    「今日挖宝次数达 X 次」
      → 完成奖励：全员明日挖宝灵石矿脉概率 +0.5%
    「今日闭关总时长达 X 小时」
      → 完成奖励：全员明日灵脉加成临时 +2%

规则：
  - 任务进度实时更新到链上（emit 事件，The Graph 索引）
  - 任务完成：次日领取奖励时自动发放
  - 未完成：当日结束自动消失，不累计到次日

博弈价值：
  → 即使是挂机玩家，今天多闭关一次也能帮全员完成任务
  → 宗主可以链上查看谁在贡献 → 踢人有依据
  → 任务类型驱动多样化活动：不是只知道 PK，也要挖宝、闭关
```

### 机制 3：宗门战志愿制

```
发起宗门战后进入「备战期」（12 小时）：

  1. 宗主/长老调用 initiateWar(targetSectId, wagerAmount, type)
     → 将 wagerAmount 从宗门金库中冻结
  2. 备战期内，所有成员调用 volunteerForWar() 表示愿意出战
     → 报名是公开的（链上可见）
  3. 备战期结束，宗主从报名名单中选 5 人（链下决策，不上链）
     → 选人名单直接合并进 commit hash，reveal 前完全保密

  报名不足 5 人时：
     → 宗主可从「未报名成员」中强制征召，补足 5 人
     → 强制出战不影响贡献度计算（但宗主记录在案）
     → 若宗门总成员数 < 5，则无法发起宗门战（合约前置检查）

  4. 双方分别 commit 出场顺序 hash
  5. 双方 reveal → 5 场 1v1 按序进行 → 结算

内部博弈：
  → 成员竞争「被选中出战」→ 出战贡献度高 → 晋升机会
  → 宗主选人需要平衡：五行搭配 vs 战力 vs 忠诚度
  → 一旦选人公布，其他成员可以分析策略
  → 高境界成员未报名 = 公开的懒惰记录 → 宗主踢人依据
```

## 问题三：宗门战的动机

### 核心原则

```
宗门战 ≠ 练级打卡
宗门战 = 有具体利益诉求的对抗

打赢有收益，拒绝有损失，打输也要付代价。
→ 纯粹的挑战 + 被挑战机制，让宗门战常态化
```

### 动机 1：主干灵脉争夺

```
全服共有固定数量「主干灵脉」（初始 = 宗门数量 × 1.5，确保有竞争稀缺）

主干灵脉价值：
  控制灵脉的宗门，全体成员在普通宗门灵脉加成之上额外 +5% 修炼速度
  每个宗门最多控制 2 条主干灵脉

抢夺方式：
  发起「灵脉争夺战」（宗门战的特殊类型）
  → 胜方获得灵脉控制权，败方失去
  → 灵脉控制权链上公开（The Graph 实时展示灵脉归属地图）

博弈：
  → 弱小宗门可以绕开强大宗门，去抢其他弱宗门的灵脉
  → 强宗门 2 条灵脉 = 成员修炼速度持续领先 → 成长雪球效应
  → 没有灵脉的宗门 = 长期竞争劣势 → 必须打出去
  → 拒绝灵脉争夺战的成本：赌注的 50%（高于普通宗门战）

灵脉归属一览（链上公开）：
  → 全服所有人可以看到当前哪条灵脉被哪个宗门控制
  → 强者多占灵脉 = 公开的不平等 → 激励其他宗门联合讨伐
```

### 动机 2：宗门金库赌注

```
宗门战赌注来自宗门金库（不是宗主个人）：

  挑战方：从金库冻结赌注
  防守方：接受挑战时，从己方金库冻结相同金额
  胜方：获得对方赌注，进入己方金库
  败方：赌注损失

宗门金库来源：
  成员捐赠（自愿，换贡献度）
  宗门战胜利所得
  （Phase 2）灵矿收益

金库损失的连锁影响：
  → 宗门金库减少 → 宗门商店补贴减少 → 渡劫丹折扣幅度降低
  → 宗门商店补贴减少 → 长期成员收益受损 → 成员可能出走
  → 成员出走 → 宗门实力削弱 → 下一场宗门战更弱
  → 形成真实的「失败代价链」

治理压力：
  → 宗主频繁输宗门战 → 金库亏损 → 成员可见 → 成员用脚投票（离开）
  → 宗主需要赢得信任 → 内部民主机制在 AI Agent 层自然涌现
```

### 动机 3：赛季积分排名

```
每 30 天一个赛季，赛季积分独立统计：

积分来源：
  赛季内宗门战胜利：  +100 分
  赛季内宗门战参与：  +10 分（败也有分，激励参与）
  成员 PvP 胜场：     +1 分/场（全宗门每日上限 100 分）
  成员渡劫成功：      +30 分
  宗门任务完成：      +5～20 分

赛季末 Top 3 奖励：
  🥇 第一名：下赛季全员灵脉加成临时 +5%（持续 30 天）
  🥈 第二名：下赛季渡劫丹折扣追加 -10%（叠加商店原折扣）
  🥉 第三名：宗门金库灵石补贴（从游戏金库按比例分配）

惩罚（防止摆烂）：
  连续 2 个赛季宗门战 0 胜场 → 90 天内无法招募新成员

→ 赛季制给宗门战制造时间节奏（赛季末期最激烈）
→ 赛季结果链上永久记录 = 宗门荣誉历史
→ 历史胜率是吸引新成员加入的公开信号
```

### 动机 4：拒绝挑战有代价

```
防守方选择拒绝时：

  标准宗门战（普通赌注）：
    拒绝 → 损失赌注的 20%（从金库扣，进游戏金库）
    → 比打输（100% 赌注）代价小，但不能无代价躲避

  灵脉争夺战：
    拒绝 → 损失赌注的 50%
    → 灵脉价值高，强迫防守方不得不接受挑战

设计意图：
  → 强宗门连续被多个弱宗门挑战 → 即使拒绝也金库流血
  → 弱宗门联合策略：多宗门同时向一个强宗门发起挑战
    → 强宗门接不完（每次宗门战都要备战/选人/commit/reveal）
    → 拒绝多次 → 金库损失累积
  → 这种策略使弱者联合具有实际价值

安全约束（防止"弱势碰瓷"Exploit）：
  合约在 challengeSect 入口强制要求：
    赌注上限 = min(攻方金库余额, 守方金库余额)
  → 防守方永远有能力匹配赌注（不存在"金库不足被强判拒绝"的情况）
  → 弱宗门无法用天量赌注套出强宗门的拒绝罚款
  → 最小赌注下限（MIN_WAGER）防止 0 成本骚扰式挑战

  频率限制（防止低成本围殴刷损）：
    同一宗门在 24 小时内最多被挑战 3 次（含灵脉争夺战）
  → 弱者联合仍可生效（3 个宗门轮流挑战 = 持续施压）
  → 但阻止同一宗门被无限次挑战导致金库瞬间清零

  灵脉战额外约束（防止零成本消耗攻击）：
    1. 攻方宣战费：赌注 × 10%（不退还，进游戏金库 50/25/25）
       → 攻方每次发起灵脉战都有实际成本，不再是零损失
       → 赌注 5000 LS → 宣战费 500 LS → 连续骚扰代价不菲
    2. 灵脉保护期：灵脉被挑战后（无论接受/拒绝/超时），该灵脉进入 7 天保护期
       → 保护期内该灵脉不可再被任何宗门挑战
       → 不影响其他灵脉的争夺
    3. 攻方冷却：同一宗门发起灵脉战后 48 小时内不可再次发起灵脉战
       → 防止单个宗门连续发起多场消耗战

  设计验证（Griefing 攻击成本分析）：
    攻击场景：光脚宗门对持有灵脉的宗门发起连续灵脉战
    旧方案（无宣战费+无冷却）：攻方零成本，守方每次被扣赌注 50% → 金库被迅速抽干
    新方案：
      攻方每次宣战费 = 赌注 × 10%（不退还）
      灵脉保护期 7 天 → 每条灵脉每 7 天最多被挑战 1 次
      攻方冷却 48h → 同一宗门每周最多发起 3~4 次灵脉战
      → 攻方要消耗守方金库，自己也要持续投入宣战费
      → 宣战费/拒绝罚比例：攻方付 10% vs 守方付 50% → 攻方仍有压力优势但不再零成本
```

## 宗门战流程（完整版）

```
第一步：发起挑战
  前置检查（合约强制）：
    require(sect.memberCount >= 5)（不足 5 人无法发起）
    require(wagerAmount <= defenderSect.treasury)（赌注不得超过防守方金库余额）
    require(wagerAmount <= attackerSect.treasury)（挑战方自身也要付得起）
    require(wagerAmount >= MIN_WAGER)（防止 0 灵石骚扰挑战，MIN_WAGER = 100 灵石）
    require(defenderChallengesIn24h < 3)（所有类型宗门战每 24h 被挑战上限 3 次）
  挑战方宗主/长老调用：challengeSect(targetSectId, wagerAmount, warType)
  warType: STANDARD（普通）/ VEIN（灵脉争夺，需指定目标灵脉 ID）
  → 从金库冻结 wagerAmount
  → 被挑战方收到链上通知

第二步：防守方决策（24 小时内）
  A. 接受：acceptChallenge(warId) → 从己方金库冻结等额赌注
  B. 拒绝：rejectChallenge(warId) → 立即结算拒绝罚款（20%/50%）
  C. 超时未响应：视为拒绝，自动扣罚款

第三步：备战期（接受后 12 小时）
  - 所有成员可调用 volunteerForWar(warId) 报名
    → 合约记录每位报名成员当时的境界（公开信息，供宗主选人参考）
    → 报名后战力 hash 仍可正常更新（换装备、闭关结束等日常操作不受限）
  - 宗主/长老从报名名单中选定 5 人（链下决策，不上链）

第四步：commit 阶段（6 小时）
  - 宗主将选人名单 + 出场顺序一并 commit（选人与排序合并为一步）
  - 双方各自 commitFighterOrder(warId, bytes32 hash)
  - hash = keccak256(出战成员地址顺序 + 32 字节随机 padding + 盐)
  - 选人名单在 reveal 前完全保密（链上只有 hash，无法反推）
  - 备战期至 commit 阶段，成员仍可进行日常操作（渡劫、装备更换等）
    → 这是合法的赛前备战行为，宗主选人时应基于成员境界和历史战绩估算潜力

  防穷举措施（32 字节随机 padding）：
    → 小宗门 5 人全员出战时仅 5! = 120 种排列，对手可穷举所有 hash
    → 加入 32 字节（256 bit）随机 padding 后，每种排列有 2^256 种 hash
    → reveal 时提交 (address[5] fighters, bytes32 padding, bytes32 salt)
    → 合约验证 keccak256(fighters ++ padding ++ salt) == commitHash
    → padding 不影响战斗结果，纯粹用于防止 hash 碰撞穷举

第五步：reveal 阶段（双方 commit 后同时 reveal）
  - 双方分别 revealFighterOrder(warId, address[5], bytes32 padding, bytes32 salt)
  - 合约验证 hash 匹配 → 依次执行 5 场 1v1
  - 战力以 reveal 时刻每位成员的最新个人战力 hash 为准（非选人时快照）
    → 设计意图：宗门战期间提升战力是对宗门的正向贡献，不应被禁止
    → 信息不对称仍然成立：对手无法预知出战名单，更无法预知成员赛前的临时提升

第六步：逐场战斗结算（全确定性）
  - 每场 1v1 = 双方对应位置出战成员比战力（含五行克制、灵兽、装备）
  - 先赢 3 场者宣告宗门战胜利

第七步：最终结算
  - 胜方金库：获得对方赌注（扣除 5% 手续费 → 游戏金库，与 PK 统一）
  - 出战成员贡献度：+10 贡（胜利方额外 +15 贡）
  - 赛季积分：胜方 +100，败方 +10
  - 灵脉争夺战：同步转移灵脉控制权（链上更新）
```

## 宗门金库

```
宗门金库（Sect Treasury）链上公开：

来源：
  成员捐赠灵石（自愿，折算贡献度）
  宗门战赌注胜利所得
  （Phase 2）灵矿产出折算

支出：
  宗门战赌注（冻结/损失）
  宗门商店补贴（折扣差价）
  宗门任务奖励（按任务完成自动发放）

治理（MVP）：
  宗主有完整金库操作权限
  所有支出链上记录 → 成员可查看金库历史流水
  成员若不满 → 只能离开（在链上产生「人员流失」记录）

→ 金库透明 = 防止宗主卷款跑路
→ 成员可以查宗门战绩 + 金库健康度后再决定是否加入

金库触底保障（防死亡螺旋）：
  触发条件：金库余额 < 100 灵石

  自动响应：
    1. 宗门商店折扣暂停（停止补贴出血，渡劫丹恢复原价）
    2. 每日系统补贴 10 灵石进金库（最低运营保障）
       → 补贴在金库 > 500 灵石后自动停止
       → 补贴来源：游戏金库（从基金会份额拨付）
    3. 宗门战赌注上限锁定为 MIN_WAGER（100 灵石），直到金库恢复

  主动退出机制：
    宗主可发起「解散投票」— 需 2/3 成员同意
    → 解散后金库余额按贡献度比例退还成员
    → 宗门名称、历史战绩链上永久保留（不可复用名称）
    → 成员即时解除宗门关系，无冷却

  设计意图：
    → 不让宗门因连续战败而「不可挽救」
    → 暂停折扣 = 止血，系统补贴 = 最低生存线
    → 但不给过多补贴 → 仍需成员捐赠和赢战来恢复
    → 解散是有尊严的退出，而非被动等死
```

## 信息公开 vs 隐藏

```
宗门公开信息（The Graph 实时展示）：
  - 宗门名称、等级、宗主地址、成员数
  - 成员列表（地址 + 职级 + 贡献度排序）
  - 宗门战历史战绩（胜/负/赌注金额）
  - 当前灵脉控制状态（哪条灵脉归谁）
  - 宗门金库当前余额
  - 当前赛季积分排名

宗门战隐藏信息（commit-reveal 保护）：
  - 出战名单（备战期内不公开）
  - 出场顺序（commit 阶段保护）

战力时效性说明：
  - 结算时采用每位成员当前最新的战斗属性（而非选人时快照）
  - 成员在备战/commit 期间进行的日常操作（渡劫、装备更换、闭关结束）
    均会更新其战斗属性，并自动生效于宗门战结算
  - 每场 1v1 结算 = 链上透明计算（与普通 PK 相同的攻防对抗公式）

博弈层次：
  → 对手宗门可以从链上看到你有哪些成员（公开），估算整体实力
  → 但不知道谁会出战 → 选人博弈
  → 不知道出场顺序 → 盲排博弈
  → 不知道每个人真实战力（且战力可能在赛前最后时刻变化）→ 信息不对称三层叠加
```

## 合约接口

```solidity
// Sect.sol — 宗门核心合约
contract Sect {
    enum WarType { STANDARD, VEIN }

    struct SectInfo {
        string name;
        address master;
        uint8 level;            // 1-4
        uint256 totalPoints;    // 永久积分（决定等级）
        uint256 seasonPoints;   // 赛季积分（每 30 天重置）
        uint256 treasury;       // 金库余额（灵石）
        uint256[] controlledVeins; // 控制的灵脉 ID 列表
    }

    struct Membership {
        uint256 sectId;
        uint8 rank;             // 0=外门, 1=内门, 2=长老, 3=宗主
        uint256 contribution;   // 贡献度
        uint256 joinedAt;
    }

    // 宗门管理
    function createSect(string name) external;        // 化神期 + 开宗令 + 1000 灵石
    function joinSect(uint256 sectId) external;
    function leaveSect() external;
    function promoteMember(address member, uint8 newRank) external; // 宗主调用
    function kickMember(address member) external;     // 宗主/长老

    // 每日领取
    function claimDailyReward() external;

    // 宗门商店
    function buyFromSectShop(uint256 itemId, uint256 qty) external;

    // 贡献度（内部 + 外部合约调用）
    function addContribution(address member, uint256 amount) external; // 仅授权合约

    // 宗门金库
    function donateToTreasury(uint256 amount) external; // 成员捐赠，+贡献度
    function getTreasuryBalance(uint256 sectId) external view returns (uint256);

    // 宗门任务
    function getSectMissions(uint256 sectId) external view returns (Mission[3] memory);
    function checkMissionProgress(uint256 sectId) external view returns (uint256[3] memory);

    // 宗门战
    // wager 必须满足：MIN_WAGER <= wager <= min(攻方金库, 守方金库)
    function challengeSect(uint256 targetSectId, uint256 wager, WarType warType, uint256 veinId) external;
    function acceptChallenge(uint256 warId) external;
    function rejectChallenge(uint256 warId) external;
    function volunteerForWar(uint256 warId) external; // 成员报名出战
    // 选人不单独上链，与出场顺序合并进 commit（防止明文泄露名单）
    function commitFighterOrder(uint256 warId, bytes32 orderHash) external; // hash = keccak256(address[5] + padding + salt)
    function revealFighterOrder(uint256 warId, address[5] fighters, bytes32 padding, bytes32 salt) external;

    // 灵脉
    function getVeinControl(uint256 veinId) external view returns (uint256 sectId);
    function getAllVeins() external view returns (Vein[] memory);

    // 赛季
    function getSeasonRanking() external view returns (SectRank[] memory);
    function claimSeasonReward(uint256 sectId) external; // 赛季结算后领取

    // 查询
    function getSectInfo(uint256 sectId) external view returns (SectInfo memory);
    function getMembership(address cultivator) external view returns (Membership memory);
    function getCultivationBonus(address cultivator) external view returns (uint256); // 灵脉加成倍率

    event SectCreated(uint256 indexed sectId, address indexed master, string name);
    event MemberJoined(uint256 indexed sectId, address indexed member);
    event MemberLeft(uint256 indexed sectId, address indexed member);
    event SectWarInitiated(uint256 indexed warId, uint256 attackerSectId, uint256 defenderSectId, uint256 wager);
    event SectWarRejected(uint256 indexed warId, uint256 penalty);
    event SectWarCompleted(uint256 indexed warId, uint256 winnerSectId, uint256 prize);
    event VeinControlChanged(uint256 indexed veinId, uint256 oldSectId, uint256 newSectId);
    event SeasonEnded(uint256 season, uint256 top1SectId, uint256 top2SectId, uint256 top3SectId);
}
```
